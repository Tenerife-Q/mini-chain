# 工业级 Rust 错误处理与容错指南：告别 unwrap 陷阱

这篇文档总结了我们在本地区块链项目中，如何将“玩具级”的报错闪退代码，升级为**“PingCAP 工业级”**的高可用代码。

## 1. 过去的陷阱：被滥用的 `unwrap()` 和 `expect()`
在初学 Rust 或者编写快速原型时，我们常常在返回 `Result`（非黑即白）或 `Option`（有或无）的函数后面直接加上 `.unwrap()`。
- **它是怎么运作的**：如果前方是 `Ok(值)`，它把值剥离出来给你；但如果前方是 `Err(错误)`，它会直接触发 `panic!`，导致当前线程（甚至整个节点程序）瞬间崩溃。
- **生产环境的危害**：对于区块链节点这简直是灾难。黑客只需要发送一个格式错误的 JSON，或者由于操作系统的 NTP 时间回拨（导致时间倒流计算报错），整个服务器就会被合法地“击杀”停机。这就是经典的 **DoS（拒绝服务）物理漏洞**。

## 2. 替代方案 A：温和降级 `unwrap_or_default()`
当我们遇到不致命的底层报错，且业务允许使用默认值时使用。
- **我们的实战**：在 `Block::new` 中获取时间戳。
- **原代码**：`.duration_since(UNIX_EPOCH).unwrap().as_secs()`
- **新代码**：`.duration_since(UNIX_EPOCH).unwrap_or_default().as_secs()`
- **好处**：当发生时间回拨异常时，`unwrap_or_default()` 发现是个 `Err`，它不会崩溃，而是返回这个类型的“默认值”。对于 `Duration`（时间段）来说，默认值就是 `0`。节点成功活了下来，只是把发生错乱的区块时间暂时标为 0。
- **启发**：你也可以使用 `unwrap_or(指定值)` 来手动指定当报错时应该拿什么值顶替。

## 3. 替代方案 B：优雅甩锅大法 `Result<T, E>` 与 `?` 操作符
当底层的工具函数（如文件读写）发生错误时，底层工具不应该自杀（不应该 `expect`），而应该把错误**上报**给调用者（甩锅）。
- **我们的实战**：在 `blockchain.rs` 的 `load_from_disk` 与 `save_to_disk` 中。
- **新语法**：
  ```rust
  pub fn save_to_disk(&self, path: &str) -> Result<(), Box<dyn std::error::Error>> {
      let serialized = serde_json::to_string_pretty(&self)?; // 注意结尾的问号
      fs::write(path, serialized)?;
      Ok(())
  }
  ```
- **核心原理解析**：
  1. 返回值变成了 `Result` 枚举。`Box<dyn std::error::Error>` 代表“无论发生的是 JSON 解析错，还是文件无权限错，都能打包成一个通用的 Error 盒子”。
  2. `?` 语法糖的本质是一个隐式的 `match`。如果 `fs::write` 成功，继续往下走；如果失败了，`?` 会立刻执行 `return Err(错误)`，提前结束当前函数，把这口黑锅原封不动地抛给上层的 `main`。

## 4. 替代方案 C：顶层兜底与降级恢复 (`match` 与 `if let`)
底层函数把故障随着 `?` 一层层冒泡抛上来之后，必定要在程序的入口（如 `main` 函数）进行最终裁决。
- **我们的实战**：
  - 代码一：`if let Err(e) = blockchain.save_to_disk(...) { ... }`
    - **原理**：我只关心存盘是不是报错了，至于 `Ok` 里面装了什么我不关心，只处理异常做个告警打印而已，节点依然全速运行。
  - 代码二：`match Blockchain::load_from_disk(...) { Ok(chain) => ..., Err(e) => ... }`
    - **原理**：经典的岔路口。如果读盘成功，拿到变量正常运转；如果遇到篡改文件，仅仅作为事件打印在控制台，防止节点“读取毒药立刻暴毙”。

## 5. 架构师启发：未来开发的通用模板
以后写 Rust 的企业级模块结构，请将以下思维融入肌肉记忆：
1. **模块内部逻辑 (Entity/Service)**：能不 `unwrap()` 就不写。把函数的返回签名改成 `Result`，内部凡事皆 `?`。只负责产生或者传递错误。
2. **边缘接口/降级情况**：实在无关紧要的参数，大胆使用 `.unwrap_or(安全值)` 蒙混过关。
3. **顶层/网关/UI层 (Controller/Main)**：在最核心的根基处，必须安排 `match` 来接住所有 `Result`，并根据错误类别，给用户友好的提示或者执行备用方案日志，确保进城的数据再毒，也毒不死系统。
